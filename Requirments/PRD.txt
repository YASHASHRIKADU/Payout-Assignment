📄 PRODUCT REQUIREMENTS DOCUMENT (PRD)
🏦 Product: Playto Payout Engine (Mini Version)
1. 🧩 Product Overview

The Playto Payout Engine is a backend system that enables merchants (freelancers/agencies) to:

View their account balance
Request payouts to their bank accounts
Track payout status

The system simulates a cross-border payment flow, where:

Customers pay in USD (simulated as credits)
The system maintains merchant balances
Merchants withdraw funds in INR (payouts)
2. 🎯 Product Goals
Ensure accurate and consistent money handling
Prevent double spending due to concurrency
Guarantee idempotent API behavior
Maintain a strict payout lifecycle
Simulate real-world asynchronous bank processing

5. ⚙️ Core Features
5.1 🧾 Merchant Ledger
Description:

Tracks all financial transactions for a merchant.

Functional Requirements:
Store transactions as:
credit (incoming funds)
debit (payouts)
Amount must be stored in paise (integer)
Balance must be derived, not stored
Formula:
Balance = SUM(credits) - SUM(debits)
5.2 💸 Payout Request API
Endpoint:
POST /api/v1/payouts
Input:
amount_paise
bank_account_id
Header: Idempotency-Key
Functional Requirements:
Validate sufficient balance
Hold funds immediately
Create payout with pending status
Return same response for duplicate requests
5.3 🔁 Idempotency
Description:

Ensures duplicate API calls do not create duplicate payouts.

Requirements:
Idempotency key must be unique per merchant
Duplicate key → return same response
Keys expire after 24 hours
5.4 🔄 Payout Processing Engine
Description:

Processes payouts asynchronously using a background worker.

Flow:
pending → processing → completed / failed
Simulation Rules:
70% → success
20% → failure
10% → stuck
5.5 🔁 Retry Mechanism
Trigger:
Payout stuck in processing > 30 seconds
Rules:
Retry up to 3 times
Exponential backoff
After max retries → mark failed and refund
5.6 🧠 State Machine
Valid Transitions:
pending → processing
processing → completed
processing → failed
Invalid Transitions:
completed → pending
failed → completed
any backward movement
Requirement:
Must be strictly enforced
5.7 🖥️ Merchant Dashboard
Features:
Display:
Available balance
Held balance
Show:
Transaction history
Payout history
Provide:
Payout request form
Show live payout status updates
6. ⚠️ Critical Constraints
6.1 💰 Money Integrity
Use integer (paise)
No floating-point values
All calculations must be DB-level
6.2 🔒 Concurrency Handling
Problem:

Simultaneous payout requests may cause overdrawing.

Requirement:
Only one request should succeed if balance is insufficient
Must use database-level locking
6.3 🔁 Idempotency
Prevent duplicate payout creation
Return consistent responses
6.4 🔄 Atomic Operations
Balance deduction + payout creation must be atomic
Refund must be atomic with failure
7. 🧱 Functional Flow
Payout Request Flow
Merchant sends payout request
System checks idempotency key
System locks merchant record
System checks available balance
If valid:
Create payout (pending)
Deduct funds (ledger debit)
Store idempotency response
Return response
Payout Processing Flow
Worker picks pending payout
Moves to processing
Simulates bank result
If success:
Mark completed
If failure:
Mark failed
Refund funds
If stuck:
Retry logic triggered
